1. Introduction

This document describes the complete data model and entity structure of the DWA Validation & Verification (V&V) System — an electronic information management system mandated by Sections 32–35 of South Africa's National Water Act (Act 38 of 1998). The system manages approximately 500,000 property records across DWS and CMA regional offices, with a target completion date of 2026.

The data layer is built on Entity Framework Core 10.0 with SQL Server 2022 as the backing store. All primary keys are Guid values configured via the Fluent API in ApplicationDBContext.OnModelCreating(). The project enables nullable reference types and implicit usings globally, which ensures compile-time safety for null handling across all entity properties.

This document covers:

  • Entity design, properties, and data types for 50+ entities across 10 domain groups
  • Relationships, foreign keys, and navigation properties as configured in the ApplicationDBContext
  • Design patterns applied: Repository, State Machine, Composite, Observer, and Chain of Responsibility
  • Data integrity rules: delete behaviours, decimal precision conventions, and PK strategy
  • Enum types, lookup/seed data, and known issues
Architecture Layer
Controllers → Services (planned) → Repositories → ApplicationDBContext → SQL Server 2022
All data access flows through repository interfaces registered in Program.cs via constructor-injected AddScoped registrations.

2. Design Patterns & Architectural Decisions

2.1 Repository Pattern

All data access is encapsulated behind interface contracts defined in the Interfaces/ directory. Each repository interface (e.g., IPropertyInterface, IFileMaster, IAddress, IForestation) declares CRUD operations that are implemented by concrete classes in Repositories/. This inversion of control enables:

  • Testability — Unit tests can substitute the EF Core InMemory provider or mock repositories
  • Separation of concerns — Controllers remain thin HTTP boundaries; business logic will live in the Services layer
  • Consistency — All new repository methods follow async/await with SaveChangesAsync()
DI Registration
Every repository pair must be registered in Program.cs:
builder.Services.AddScoped<IPropertyInterface, PropertyRepository>();

2.2 Lookup Table / Type Pattern

Rather than encoding classification values as enums in code (which require recompilation to extend), the system uses separate Type entities as extensible lookup tables. These include:

Type EntityReferenced ByPurpose
EntitlementTypeEntitlement.EntitlementTypeIdClassifies ELU entitlement categories
AuthorisationTypeAuthorisation.AuthorisationTypeIdPermit, S32/33, Transfer, GA, etc.
CustomerTypePropertyOwner.CustomerTypeIdOwner classification (Individual, Company, Trust, etc.)
CropTypeCrop.CropTypeIdCrop classification hierarchy
LetterTypeLetterIssuance.LetterTypeId7 Section 35 letter types per NWA

All type entities are seeded on application startup via SeedDataService, which checks if the table is empty before inserting. This allows DWS administrators to add new classification values through the database without requiring a code deployment.

2.3 Self-Referencing Hierarchies

Three entities in the system use self-referencing foreign keys to model tree structures. This is an application of the Composite pattern, where each node can act both as an individual element and as a container of child elements.

EntityFK PropertyNavigationHierarchy Purpose
OrganisationalUnit ParentOrgUnitId ParentOrgUnit / ChildOrgUnits National → Provincial → Regional → CMA office hierarchy
Property ParentPropertyId ParentProperty / ChildProperties Subdivision and consolidation lineage tracking
CaseComment ParentCommentId ParentComment / Replies Threaded comment replies on V&V cases

All three self-referencing relationships use DeleteBehavior.Restrict to prevent cascading deletes that would orphan child records. The nullable FK (Guid?) indicates root-level nodes (no parent).

2.4 Join Table / Explicit Many-to-Many

The system uses explicit join entities rather than EF Core's implicit many-to-many configuration. This design choice is deliberate: each join table carries payload properties beyond the two foreign keys, making implicit skip navigations insufficient.

Join EntityLeft FKRight FKPayload
PropertyOwnership PropertyId → Property PropertyOwnerId → PropertyOwner TitleDeedNumber, TitleDeedDate
PublicUserProperty PublicUserId → PublicUser PropertyId → Property ApprovedByUserId, ApprovedDate, Status
ObjectionDocument ObjectionId → Objection DocumentId → Document (Link-only; enables independent document reuse)

2.5 Polymorphic Identity

The system maintains two separate identity realms to enforce strict security boundaries between internal DWS staff and external water users:

EntityBase ClassAuth SchemePurpose
ApplicationUser Extends IdentityUser (ASP.NET Identity) Internal cookie scheme DWS officials, validators, capturers, managers
PublicUser Standalone entity (no IdentityUser) Separate external cookie scheme Water users, landowners, legal representatives

Both user types can sign documents. The DigitalSignature entity has two optional FKs: ApplicationUserId? and PublicUserId?, with exactly one populated at any time. The same pattern appears in Document (UploadedByUserId? / UploadedByPublicUserId?), Notification, and SignatureRequest.

2.6 State Machine in Data

The V&V workflow is modelled as a State pattern implemented across three cooperating entities:

State Machine Triad
WorkflowState — Defines the 31 possible states (seeded reference data), grouped by phase.
WorkflowInstance — Tracks which state a specific FileMaster case is currently in.
WorkflowStepRecord — Provides immutable history of every state transition, including who completed it and when.

WorkflowInstance.CurrentWorkflowStateId points to the active state. The WorkflowEngine (planned service) exposes CanTransition(user, fileMaster, targetState) which evaluates a chain of ITransitionGuard implementations before allowing state changes.

2.7 Immutable Audit Trail

Every significant action in the system is recorded in the AuditLog entity. This is an append-only table — rows are never updated or deleted. Each entry captures:

  • EntityType + EntityId — Generic entity reference (polymorphic by convention, not by FK)
  • OldValues / NewValues — JSON-serialised snapshots of the before/after state
  • ApplicationUserId, UserName, IPAddress, Timestamp — Full forensic context

2.8 Row-Level Security

External portal data isolation is enforced through the PublicUserProperty join table. An external user can only access case data for properties explicitly linked to their account. The link requires:

  1. The external user requests access (status = Pending)
  2. A DWS official approves the link (ApprovedByUserId, ApprovedDate, status = Approved)
  3. All external portal queries are scoped by PublicUserId via the approved PublicUserProperty records

3. Domain Group: Geography & Organisation

5 Entities

DWS operates a national-to-regional hierarchy that maps directly onto the case management access model. South Africa has 9 provinces and 19 Water Management Areas (WMAs) nationally. The organisational hierarchy flows: National DWS (Head Office, Pretoria) → Provincial DWS Offices → Regional Offices / WMAs → Catchment Areas → CMAs (Catchment Management Agencies, where established) → Service Providers / Consultants.

A FileMaster case belongs to a WMA/Regional Office, and access is scoped by the OrganisationalUnit assigned to the user. When a user is assigned to a province or WMA, they can only see and operate on cases in that scope, unless they hold a national-level role.

4. Domain Group: Property & Ownership

4 Entities

Property is the central entity of the entire system. Every V&V case (FileMaster), every water use record (Irrigation, Forestation, Storing, FieldAndCrop), every dam calculation, and every satellite image is ultimately linked to a Property. A property represents a farm or land parcel identified by its SG Code (21-digit Surveyor General code), which uniquely identifies the parcel in South Africa's cadastral system.

The Quaternary Drainage code locates the property within the hydrological catchment hierarchy, and the WMA foreign key links it to one of the 19 Water Management Areas. Properties can be subdivided or consolidated: the ParentPropertyId self-referencing FK tracks this lineage, with PropertyStatus indicating whether the property is Active, Consolidated, or Subdivided.

Property ownership is modelled as a many-to-many relationship through PropertyOwnership, which carries the title deed number and date as payload. The IsHDI flag on PropertyOwner indicates whether the owner is a Historically Disadvantaged Individual (HDI) — a classification relevant to water allocation reform.

5. Domain Group: Water Use Reference Data

9 Entities

These are the lookup and reference tables that classify water use types, periods, sources, and related institutional entities. They are seeded on application startup via SeedDataService and are designed to be extended by administrators without code changes.

The Period entity distinguishes between the qualifying period (the 2 years before NWA commencement on 1 October 1998) and the current period. The WaterSource entity combines a name with a WaterSourceType enum (SURFACE, BOREHOLE, STREAM). The qualifying period is unified across all water source types: 1 October 1996 – 30 September 1998 (per SFRA court order).

7. Domain Group: Entitlements & Authorisations

4 Entities

The Entitlement entity represents the outcome of the V&V process — the determined volume and type of Existing Lawful Use (ELU). It is the ultimate deliverable of the system: after all control points are complete and the Section 35 validation process concludes, an Entitlement record captures the confirmed water use entitlement for a property.

Authorisation records capture the supporting legal instruments that provide evidence for lawful water use: permits, Section 32/33 approvals, transfers, General Authorisations, irrigation board records, and other water act instruments. The relationship is: FileMaster → Entitlement (the determination) and FileMaster → Authorisations (the evidence supporting that determination).

8. Domain Group: Case Management

2 Entities

FileMaster is THE central case entity of the V&V system. It represents the V&V case file for a single property — the digital equivalent of the physical file folder described in Appendix A of the DWS requirements document. Every property under investigation has one (or more, in rare cases) FileMaster record that tracks the complete lifecycle from inception to ELU confirmation.

FileMaster Field Groups

Identification

RegistrationNumber (required) is the WARMS registration number. CaseNumber is the auto-generated V&V case number (business rule TBD). SurveyorGeneralCode identifies the property in the SG cadastral system. FileNumber is the physical file reference.

Catchment & Farm Details

PrimaryCatchment and QuaternaryCatchment locate the property in the hydrological hierarchy. FarmName, FarmNumber, FarmPortion, and RegistrationDivision identify the farm.

Registration Flags

Three boolean flags indicate what the property is registered for in WARMS: RegisteredForTakingWater, RegisteredForStoring, RegisteredForForestation. These determine which water use data entities must be populated.

Assignment & Status

ValidatorId and CapturePersonId FK to ApplicationUser for role-based assignment. OrgUnitId scopes the case to an organisational unit. ValidationStatusName tracks the Appendix B status. WorkflowInstanceId links to the active workflow instance for state machine tracking.

The engine exposes CanTransition(user, fileMaster, targetState) which returns a bool and reason string. 10. Domain Group: Letters, Documents & Signatures

5 Entities

Section 35 letters are the primary instrument for communicating V&V outcomes to water users. The letter lifecycle follows the Chain of Responsibility pattern: if a water user does not respond to Letter 1 (S35(1) notice), the system escalates to Letter 1A (S53(1) directive). If additional information is needed, Letter 2 is issued; non-response escalates to Letter 2A. Confirmed ELU results in Letter 3 (the ELU certificate). Unlawful use triggers Letters 4A and 4/5 (directives to cease).

12. Domain Group: Public Portal

5 Entities

External stakeholders (water users, landowners, legal representatives) access a separate portal with its own identity regime, isolated from the internal DWS staff system. The portal is implemented as a separate MVC Area (/Areas/ExternalPortal/) with its own authentication cookie scheme.

Data isolation is enforced at every level: PublicUserProperty controls which properties an external user can see, and all data queries in the external portal are always scoped by PublicUserId. An external user can never retrieve records for a property not linked to their account.

14. Data Integrity Rules

14.1 Global Delete Behaviour

The ApplicationDBContext.OnModelCreating() method applies a global DeleteBehavior.Restrict to all relationships via a loop over all foreign keys. This is a deliberate safety measure: SQL Server does not allow multiple cascade paths, and Restrict forces explicit deletion in the correct order, preventing accidental data loss.

Prior to the global override, specific relationships are configured with DeleteBehavior.SetNull where orphaning is acceptable (e.g., Property.AddressId, FileMaster.OrgUnitId, FileMaster.ValidatorId). However, the global loop at the end of OnModelCreating overrides these to Restrict as well.

Important: Global Override
The global DeleteBehavior.Restrict loop at the end of OnModelCreating overrides all individually-configured SetNull behaviours. This means that in practice, no cascade or set-null deletes occur — all deletions must be performed explicitly in the correct dependency order.

14.2 Primary Key Strategy

All primary keys are Guid (not auto-incrementing integers). This supports:

  • Distributed generation — IDs can be created client-side or in disconnected scenarios without database roundtrips
  • Merge safety — No collision risk when merging data from regional offices
  • Security — Non-sequential IDs prevent enumeration attacks

All PKs are configured explicitly in OnModelCreating() via HasKey() calls. No [Key] data annotations are used on model classes.

14.4 Nullable Reference Types

The project enables nullable reference types globally. Properties marked with ? (e.g., string?, Guid?) are genuinely optional and map to nullable SQL columns. Properties marked with required must be set at construction time and map to non-nullable SQL columns.

14.5 Fluent API Configuration

All entity configuration is done in ApplicationDBContext.OnModelCreating() using the EF Core Fluent API. No data annotations ([Key], [Required], [ForeignKey]) are used on model classes, with the exception of [Column(TypeName = "...")] for decimal precision. This centralises all database schema decisions in a single location.

15. Seed Data

The SeedDataService runs on application startup and populates reference tables using a check-if-empty pattern: for each table, it queries whether any rows exist, and only inserts seed data if the table is empty. This ensures seed data is applied on first run but never duplicated on subsequent starts.

16. Known Issues & Planned Changes

16.1 IEntitlement Interface Naming Conflict

Interfaces/IEntitlement.cs defines public interface Entitlement which conflicts with Models/Entitlement.cs. This must be renamed to public interface IEntitlement to follow the project's naming convention and resolve the ambiguity. Until fixed, the compiler may resolve references incorrectly depending on using directives.

16.2 Synchronous Methods in Older Repositories

Some existing repositories (e.g., PropertyRepository) use synchronous SaveChanges() instead of the preferred async/await pattern with SaveChangesAsync(). These should be migrated to async as part of ongoing codebase modernisation. All new repositories must use async methods.

16.3 WaterUserInput Entity

A WaterUserInput entity is planned but not yet created. This will capture structured data from water user responses to Section 35 letters, replacing the current free-text ResponseNotes field on LetterIssuance.

16.4 Global Delete Behaviour Override

The global DeleteBehavior.Restrict loop at the end of OnModelCreating currently overrides all individually-configured SetNull behaviours. This is intentionally conservative but may need refinement as the application matures. Specific SetNull relationships (e.g., removing a Validator from a case without deleting the case) should be moved after the global loop to take precedence.

16.5 Address.Province as String

The Address.Province field is stored as a plain string rather than a foreign key to the Province entity. A future migration should add a ProvinceId? FK to enable referential integrity and consistent province naming.

16.6 Enum Typos

VerifactionScenario (should be "Verification") and OBTAIN_OTHER_SUPPORTIVR_INFORMATION (should be "SUPPORTIVE") contain typos that are preserved to avoid breaking existing migrations. A coordinated rename with a migration and data fix is planned.
