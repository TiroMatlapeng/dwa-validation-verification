1. Introduction

The Validation and Verification (V&V) of existing lawful water use is a statutory obligation under Sections 32–35 of the National Water Act (Act 36 of 1998). The Department of Water and Sanitation (DWS) is required to verify the lawfulness and extent of all existing water use across South Africa, encompassing approximately 500,000 property records managed by DWS regional offices and Catchment Management Agencies (CMAs) across 19 Water Management Areas.

The DWA V&V System is an electronic information management system designed to digitise, automate, and enforce the V&V process. At its core is the Workflow Engine — a state machine that governs the lifecycle of every case file (FileMaster) from initial inception through technical evaluation, legal verification, statutory correspondence under Section 35, and final determination of Existing Lawful Use (ELU).

This document provides a comprehensive specification of the four principal workflows within the system:

  1. The V&V Workflow State Machine — the master process governing all control point transitions
  2. The Section 35 Letter Workflow — a nested state machine for statutory correspondence
  3. The End-to-End Data Flow — from external systems through processing engines to final output
  4. The Electronic Signature Flow — dual-path signing for internal officials and external water users
Governing Requirements
All workflows in this document are designed to comply with the Requirements for Verifying the Lawfulness of Existing Water Use and Extent of Existing Lawful Water Use, Edition 3, July 2024 (DWS). The target completion date for the national V&V programme is 2026.

2. Design Philosophy

State Machine Pattern

The workflow engine is built on a finite state machine pattern. Every FileMaster case exists in exactly one state at any given time, and may only transition to a defined set of successor states. This design was chosen for three fundamental reasons:

  • Deterministic: Given a current state and a transition request, the outcome is always predictable. There are no ambiguous or race-condition-prone paths through the process.
  • Auditable: Every state transition is recorded as an immutable audit entry, capturing the user, timestamp, previous state, new state, and reason. This provides a complete forensic trail for any case.
  • Enforceable: The state machine prevents out-of-order operations. A case cannot reach the Section 35 letter phase without first completing all nine control points, and each control point has verifiable preconditions.

The Guard Pattern (ITransitionGuard)

Each state transition is protected by one or more transition guards — implementations of the ITransitionGuard interface. Guards are checked in sequence before any transition is permitted. A guard may verify that required data is present (e.g., SAPWAT calculations exist for CP6), that the requesting user holds the correct role (e.g., only RegionalManager or above may approve Section 35 letters), or that business rules are satisfied (e.g., dam calculations are present or explicitly marked as not applicable).

Immutable Audit Trail

The AuditEntry table is append-only. Records are never updated or deleted. Each entry captures: the acting user, the action performed, the entity affected, the previous and new state values, a timestamp, and additional metadata. This log provides the evidentiary basis for any legal challenge to a V&V determination.

Domain Events and Notifications

Successful state transitions raise domain events that are consumed by the NotificationService. This decoupled, event-driven architecture ensures that notifications (email, in-app, SMS) are triggered automatically without the workflow engine needing direct knowledge of the notification mechanism. For example, when a Letter 1 transitions to Issued, a LetterIssuedEvent is raised, which the notification service consumes to alert the relevant water user and the case validator.

Separation of Concerns
The workflow engine is responsible solely for state management and guard evaluation. All side effects — notifications, PDF generation, audit logging, integration updates — are handled by dedicated services listening to domain events.

3. The V&V Workflow State Machine

3.1 Overview

The V&V Workflow State Machine governs the entire lifecycle of a FileMaster case. As shown in Figure 1, the process is organised into five macro phases, each containing one or more sequential states:

  1. Case Inception — Document assimilation, data staging, and readiness verification
  2. Technical Validation — Satellite/GIS analysis, volumetric calculations, and comparison logic
  3. Legal Verification — Lawfulness assessment, entitlement-versus-authorisation evaluation, and determination
  4. Statutory Procedure — Section 35 notice issuance, representation review, and finality
  5. Final Determination / Non-Compliance — ELU certification or directive issuance for unlawful use

A case begins at Create Case and advances linearly through each phase. At the Determination stage, the case may branch into Non-Compliance if unlawful use is identified, leading to Directive Issuance under Section 53(1). Otherwise, the case proceeds through the statutory procedure to Final Determination.

3.2 Case Inception Phase

The Case Inception phase establishes the evidentiary foundation for the entire V&V process. It comprises three states:

Case_Inception

The entry point. A Validator or higher creates a new FileMaster record, linking it to a Property via the SG Code and to the appropriate Water Management Area and organisational unit.

Document_Assimilation

All foundational documents are gathered and associated with the case: identity documents, title deeds, WARMS registration references obtained from eWULAAS, SG diagrams, and any prior V&V study records. The system verifies that mandatory document types are present before allowing progression.

Data_Staging

Imported data is normalised, validated, and staged for technical analysis. This includes reconciling property boundaries against SG data, confirming quaternary drainage and WMA assignments, and running a readiness check to ensure all prerequisite data sets are in place.

Transition Trigger: File Complete
The case advances from Case Inception to Technical Validation only when the readiness check confirms that all mandatory documents have been assimilated and data staging is complete. This is enforced by the FileCompleteGuard.

3.3 Technical Validation Phase

The Technical Validation phase is where the quantitative analysis of water use takes place. It comprises three states:

Satellite_GIS_Analysis

Satellite imagery from both the qualifying period (1 October 1996 – 30 September 1998, unified per SFRA court order) and the current period is analysed. The processed artefact produced from this analysis is a Mapbook; SateliteImage records store the raw imagery metadata only. Crop fields and dams are digitised using GIS tools, producing spatial layers that define the areas of water use for both periods.

Volumetric_Calculation

The Calculator Engine executes three calculation models:

  • SAPWAT — Crop water requirement modelling (output in mm/ha/a) based on crop type, irrigation system, and field area, calibrated per catchment
  • Dam Volume Calculation — Using either the wall-length method or the area method (Appendix D formulas) with appropriate shape factors
  • SFRA/Forestation — Stream flow reduction activity calculations for afforested areas, determining qualifying period hectares, current hectares, and lawful/unlawful volumes

Comparison_Logic

The system compares qualifying-period water use against current-period use, categorising volumes across the water use categories defined in the requirements (rows 4.1 through 4.9): possible existing, possible lawful, possible existing lawful, possible unlawful, registered (WARMS), present-day use, and possible unlawful increase or decrease.

Transition Trigger: Tech Sign-off
A Validator must formally sign off on the technical analysis before the case advances to Legal Verification. The guard verifies that all three calculation types have been completed (or explicitly marked as not applicable) and that comparison logic results are recorded.

3.4 Legal Verification Phase

The Legal Verification phase applies the legal framework of the National Water Act to the technical results. It comprises three states:

Lawfulness_Assessment

A legal opinion is formed on the lawfulness of the water use, considering all evidence gathered: GWCA authorisations, permits, Section 32/33 approvals, irrigation board records, water court orders, General Authorisations, and any other applicable legislative instruments. Each water use category is assessed against its legal basis.

Entitlement_vs_Authorization

The determined Existing Lawful Use (ELU) is compared against the user's current authorisation status. This comparison produces one of three outcomes:

  • Lawful — All water use is within authorised limits
  • Unlawful — Water use exceeds authorisation or lacks any legal basis
  • Possible Unlawful Increase/Decrease — A delta exists between the ELU and current use that requires further investigation

Determination

The formal determination is recorded on the Validation entity, including ELU volumes, registration status, and the classification of each water use component. This state represents the point at which the case is ready to enter the statutory procedure or, if unlawful use is identified, to branch into the non-compliance path.

Transition Trigger: S35 Initiation
The RegionalManager approves the determination, which triggers the transition into the Statutory Procedure phase. If unlawful use is identified, a parallel branch to Non-Compliance may be initiated.

3.5 Statutory Procedure Phase

The Statutory Procedure phase enacts the requirements of Section 35 of the National Water Act through formal correspondence with the water user. It comprises three states:

Section_35_1_Notice

Letter 1 — the S35(1) Notice to Apply for Verification — is generated, signed by the Regional Manager, and served to the water user. This letter must be served in person (hand delivery by an official, agent, or sheriff), and the system records the serving official, delivery date, and method.

Representation_Review

The water user's response is received and reviewed. The user may agree with the determination, disagree and provide additional evidence, or fail to respond within the statutory period. Each outcome triggers a different downstream path (detailed in Section 4).

Section_35_4_Finality

Upon resolution of all representations and any additional information requests, the Section 35(4) certificate is issued — Letter 3 — which formally confirms the extent and lawfulness of the water use. This is the terminal state for compliant cases.

Transition Trigger: Certificate Issued
The case reaches Final Determination when Letter 3 (the S35(4) ELU certificate) is signed, issued, and acknowledged. The case is then closed or archived.

3.6 Non-Compliance Branch

When a water user fails to respond to statutory notices, refuses to comply with directives, or when the determination reveals unlawful water use that cannot be regularised, the case branches into the Non-Compliance path. This branch leads to Directive_Issuance, where the following enforcement actions may be taken:

  • Letter 4A — S53(1) Notice of Intent to issue a directive to stop unlawful water use
  • Letters 4 & 5 — S53(1) Directive to stop unlawful water use

The non-compliance branch has its own letter lifecycle (detailed in Section 4.7) and terminates at Enforcement Complete, after which the case is closed with the unlawful use determination on record.

Legal Significance
Directive issuance under Section 53(1) is a formal enforcement action with legal consequences. The system must record all correspondence, delivery confirmation, and response timelines with forensic precision, as these records may be required in subsequent legal proceedings.

3.7 Transition Guards

Every state transition in the workflow is protected by one or more guards that must all pass before the transition is permitted. The WorkflowEngine.CanTransition(user, fileMaster, targetState) method evaluates all guards and returns a boolean result with a reason string. The following table lists representative guard examples:

Transition Guard Requirement
CP1 → CP2 WarmsExportGuard WARMS export date must be recorded on the FileMaster
CP1 → CP2 SatelliteImageGuard At least one SateliteImage record must exist for qualifying (1 October 1996 – 30 September 1998) and current periods; and a Mapbook must be confirmed as produced
CP2 → CP3 SpatialDataGuard River and Property SG codes must be confirmed
CP5 → CP6 GisLayerGuard GIS spatial layers for qualifying and current periods must be confirmed
CP6 → CP7 SapwatGuard FieldAndCrop records with SAPWATCalculationResult values must be present
CP7 → CP8 EluCalculationGuard Validation record with ELU volumes must be set
CP8 → CP9 DamCalculationGuard DamCalculation records must be present, or explicitly marked N/A
CP9 → S35 ForestationGuard Forestation records must be present, or explicitly marked N/A
Any letter signing RoleGuard Only RegionalManager or above may sign Section 35 letters
Any transition AuditGuard An AuditEntry must be successfully written before the transition completes

4. The Section 35 Letter Workflow

4.1 Overview

The Section 35 Letter Workflow is a nested state machine that activates after Control Point 9 (CP9) is complete. It governs the generation, signing, issuance, tracking, and response handling of all statutory letters required by Sections 35 and 53 of the National Water Act.

As shown in Figure 2, each letter follows a standard lifecycle:

  1. Generated — The PDF is created by the LetterService using QuestPDF templates
  2. Signed — An authorised signatory (RegionalManager or above) applies an electronic signature
  3. Issued — The letter is dispatched to the water user via the appropriate delivery method
  4. Awaiting — The system tracks the statutory response period
  5. Response / Escalation — The user responds (agreed/not agreed) or fails to respond, triggering the next letter in the sequence
Letter Issuance Records
Every letter transition records: IssuedDate, IssuedByUserId, SignedByUserId, DeliveryMethod, PostalRegistrationNumber (if registered post), DueDate, and ResponseReceivedDate. These fields form the evidentiary record for legal compliance.

4.2 Letter 1: S35(1) Notice to Apply

Letter 1 is the initiating notice under Section 35(1) of the National Water Act, informing the water user that their existing water use is being verified and requesting them to apply for verification. The letter lifecycle is:

Letter 1 Created → L1_Generated → L1_Signed → L1_Issued → L1_Awaiting

Personal Service Requirement
Letter 1 must be served in person — by hand delivery through a DWS official, authorised agent, or sheriff of the court. The system records the serving official's identity, the date and time of delivery, and the GPS coordinates of delivery where available. This is a non-negotiable legal requirement under the NWA.

The Regional Manager (or delegated authority) signs the letter electronically before issuance. After issuance, the system enters the L1_Awaiting state and monitors three possible outcomes:

  • L1_Agreed — The water user responds within the statutory period and agrees with the verification. The case advances to Letter 3 (ELU confirmation).
  • L1_NotAgreed — The water user responds but disagrees. This may trigger Letter 2 (request for additional information) or proceed to further review.
  • No Response — The statutory period expires without a response. The case escalates to Letter 1A.

Additionally, a Return to Sender branch handles undeliverable letters: the state transitions to L1_RTS, and the letter may be reissued with an updated address or alternative delivery method.

4.3 Letter 1A: S53(1) Directive

Letter 1A is an escalation under Section 53(1) — a formal directive issued when the water user fails to respond to Letter 1. It follows the same lifecycle as Letter 1 (Created → Generated → Signed → Issued → Awaiting) and carries stronger legal force. The directive compels the water user to apply for verification within a specified period, with consequences for continued non-compliance.

Upon response to Letter 1A (L1A_Responded), the case returns to the standard correspondence flow. If no response is received, the case may escalate to enforcement (Letters 4A and 4&5).

4.4 Letter 2: S35(3)(a) Request for Additional Information

Letter 2 is issued under Section 35(3)(a) when the responsible authority requires additional information to complete the verification. This typically arises when:

  • The water user disputes the technical findings and provides counter-evidence that requires evaluation
  • Documentary gaps exist (missing title deeds, permits, or historical records)
  • The GIS analysis reveals anomalies that require field verification by the water user

The letter specifies exactly what information is required and sets a deadline for submission. Upon receipt of the response (L2_Responded), the additional information is incorporated into the case and the technical evaluation may be revisited.

4.5 Letter 2A: S35(1) Directive for Information

Letter 2A is the enforcement escalation for Letter 2 — issued under Section 35(1) as a formal directive when the water user fails to provide the additional information requested in Letter 2 within the statutory period. It follows the standard letter lifecycle and carries the same legal weight as Letter 1A.

4.6 Letter 3: S35(4) ELU Confirmation

Letter 3 is the culmination of the V&V process for compliant cases — the Section 35(4) certificate that formally confirms the extent and lawfulness of the water user's existing water use. This letter is signed by the responsible authority and constitutes an official determination under the National Water Act.

The issuance of Letter 3 leads to one of two outcomes:

  • ELU Confirmed — All water use is determined to be lawful. The case transitions to Case Complete and is archived.
  • Unlawful Use Identified — The determination reveals water use that exceeds the ELU or lacks legal authorisation. The case branches to enforcement (Letters 4A and 4&5).
The ELU Certificate
Letter 3 is the single most important output of the V&V process. It provides legal certainty to the water user regarding their existing lawful use and forms the basis for future water use licensing decisions. The system must ensure this letter is generated with complete accuracy, properly signed, and securely stored.

4.7 Letters 4A and 4&5: Enforcement

When unlawful water use is identified and the water user fails to regularise their use, the enforcement sequence is initiated:

Letter 4A — S53(1) Notice of Intent

A formal notice informing the water user that the responsible authority intends to issue a directive to stop unlawful water use. This provides a final opportunity for the user to comply voluntarily or to make representations before enforcement action is taken.

Letters 4 & 5 — S53(1) Directive to Stop

The final enforcement instrument — a formal directive under Section 53(1) ordering the water user to cease all unlawful water use. Upon issuance and acknowledgement (or expiry of the response period), the case transitions to Enforcement Complete.

4.8 eWULAAS Integration

Upon final determination — whether through ELU confirmation (Letter 3) or enforcement completion (Letters 4&5) — the system pushes the determination data back to eWULAAS via the IntegrationService.UpdateWarmsData() method. This ensures that the national water use registration system reflects the verified and validated water use information.

WARMS Integration Removed
Direct integration with the legacy WARMS system has been removed from the architecture. All water use registration data now flows exclusively through eWULAAS, which serves as the single authoritative source for registration information. The V&V system reads from eWULAAS at case inception and writes back to eWULAAS at case closure.

5. End-to-End Data Flow

5.1 System Architecture Overview

The DWA V&V System is structured as a four-tier architecture, as illustrated in Figure 3:

  1. External Systems — Upstream data sources that feed into the V&V process
  2. Application Layer — The V&V phases, processing engines, and service layer
  3. Data Layer — The repository layer and SQL Server database
  4. Outputs — The internal DWS application and the public-facing external portal

Data flows from left to right through the system: external source data is ingested via the Integration Service, processed through the three V&V phases using specialised engines, persisted via the repository layer to SQL Server, and presented through both the internal ASP.NET Core MVC application and the separate external portal.

5.2 External System Integration

The V&V System integrates with four external data sources, all managed through the IntegrationService:

External System Data Provided Integration Method
eWULAAS Water use registration data (WARMS records), registration status, licence information Primary integration — API-based read/write
Surveyor General / DRDLR SG farm boundaries, cadastral data, SG diagrams, property extent information Spatial data import
Deeds Office / DRDLR Title deed records, property ownership history, transfer records Document retrieval
Satellite / NoSQL Store Satellite imagery for qualifying period (1996–1999) and current period, stored in a NoSQL document store for high-volume binary data Blob/document storage integration
eWULAAS as Primary Integration Point
eWULAAS (the electronic Water Use Licence Application and Authorisation System) is the primary external system. It replaces direct WARMS integration and serves as the single source of truth for water use registration data. Both initial data retrieval (at case inception) and final determination updates (at case closure) flow through eWULAAS.

5.3 The Three V&V Phases

The application layer processes data through three sequential phases, each corresponding to a section of the DWS requirements document:

Phase 1: Inception

The foundational phase that establishes the case and gathers all prerequisite data:

  • WARMS Audit — Obtain the latest registration data from eWULAAS and audit for new records, ownership changes, and data discrepancies
  • GIS Identification — Identify unregistered water users through GIS analysis of current satellite imagery
  • Database Analysis — Assign record status to all properties using the validation status categories (Not Commenced, In Process, Completed, etc.)
  • Inception Report — Draft and approve the project inception report documenting scope, methodology, and findings
  • Public Participation — Conduct WMA-level public participation sessions (Session 1) to launch the V&V process in the area

Phase 2: Validation (Technical Extent)

The technical analysis phase where water use is quantified and classified against the qualifying period (1 October 1996 – 30 September 1998, unified per SFRA court order):

  • Evaluate WARMS Data — Review registered volumes, crops, hectares, and irrigation methods against field evidence
  • Spatial Information — Obtain and verify SG farm boundaries, catchment boundaries, and river data
  • Additional Information — Gather permits, Section 32/33 approvals, transfers, General Authorisations, and other legislative instruments
  • GIS Analysis & Field Digitising — Digitise crop fields and dams for both qualifying and current periods
  • SAPWAT Modelling — Calculate crop water requirements per field using the SAPWAT model
  • ELU Calculation — Determine lawful, unlawful, and possible unlawful increase/decrease volumes
  • Dam Volume Calculation — Calculate storage capacity using wall-length or area methods
  • SFRA Calculation — Determine stream flow reduction for afforested areas

Phase 3: Validation

The statutory correspondence and closure phase:

  • Section 35 Letters — Generate, sign, issue, and track all statutory correspondence (Letters 1 through 5)
  • Response Tracking — Monitor statutory deadlines and process water user responses
  • Case Closure — Issue the ELU certificate or enforcement directive, update eWULAAS, and archive the case

5.4 The Engine Layer

The engine layer comprises six specialised components that provide the core processing capabilities of the system. These engines are coordinated through a mediator pattern, with domain events providing loose coupling between components:

Engine Responsibility Key Technologies
Calculator Engine Executes SAPWAT crop water requirement modelling, dam volume calculations (Appendix D wall-length and area methods), and SFRA stream flow reduction calculations Domain-specific calculation logic; shape factors (0.33, 0.4, 0.5)
Letter Engine Generates Section 35 letter PDFs from templates, manages letter lifecycle states, and tracks delivery and response QuestPDF for PDF generation; letter type templates
Workflow Engine Enforces the state machine, evaluates transition guards, manages control point progression, and raises domain events on transitions State machine pattern; ITransitionGuard interface
Notification Engine Delivers email, in-app, and SMS notifications triggered by domain events (state transitions, letter issuance, deadline reminders) Email provider integration; in-app notification hub
Signature Engine Captures electronic signatures via SignaturePad.js, generates SHA-256 hashes, embeds signatures into PDFs, and verifies document integrity SignaturePad.js; X.509 server-side verification
Audit Engine Writes immutable audit log entries for every state transition, data modification, and significant user action Append-only AuditEntry table; structured logging

The engines communicate through domain events following the mediator pattern. When the Workflow Engine completes a state transition, it publishes a StateTransitionEvent. The Notification Engine subscribes to this event and dispatches the appropriate notifications. The Audit Engine subscribes to all events and logs them. The Letter Engine listens for transitions into letter-related states and initiates PDF generation. This decoupled architecture ensures that each engine can evolve independently without creating tight dependencies.

5.5 Service and Repository Layers

The Service Layer orchestrates business logic by coordinating between engines, repositories, and external integrations. Services are injected via ASP.NET Core dependency injection and follow the pattern established by the existing PropertyService stub. Each service encapsulates a bounded context of domain logic (e.g., LetterService manages all letter-related operations, ValidationService manages ELU calculations).

The Repository Layer provides data access through Entity Framework Core, implementing repository interfaces defined in the Interfaces/ directory (e.g., IPropertyInterface, IAddress, IForestation). All new repository methods follow the async/await pattern with SaveChangesAsync(). The single ApplicationDBContext manages all entity relationships, with primary keys and foreign keys configured explicitly in OnModelCreating().

Repository Pattern and Interface Contracts
Every data access operation is abstracted behind an interface, allowing the service and engine layers to be tested independently of the database. New repositories must be registered in Program.cs using builder.Services.AddScoped<IXxx, XxxRepository>().

5.6 Database Schema Groups

The SQL Server database (dwa_val_ver) is organised into five logical schema groups, as reflected in the data flow diagram:

Schema Group Tables Purpose
Core Domain Property, FileMaster, Entitlement, Validation, Irrigation, FieldAndCrop, DamCalculation, Forestation, Storing, PropertyOwner, PropertyOwnership The central V&V entities representing properties, case files, water use data, and calculations
Organisation & Reference Province, WaterManagementArea, OrganisationalUnit, River, Address, SateliteImage, lookup tables (ValidationStatus, RegistrationStatus, etc.) Organisational hierarchy, geographic reference data, and seed/lookup values
Workflow Schema WorkflowState, WorkflowInstance, WorkflowStepRecord, AuditEntry State machine state storage, transition history, and immutable audit log
Correspondence LetterType, LetterIssuance, UploadedDocument, DigitalSignature, SignatureRequest Section 35 letter types, issuance records, document storage references, and electronic signatures
Portal Schema ExternalUser, ExternalUserProperty, CaseComment, CaseNotification, Protest, ProtestDocument External water user accounts, case comments, notifications, and protest records for the public-facing portal

All primary keys are Guid values, configured in OnModelCreating(). Decimal columns use decimal(18,2) for volumes and areas, and decimal(9,6) for geographic coordinates (latitude/longitude).

6. Electronic Signature Flow

6.1 Overview

The electronic signature flow provides a dual-path signing mechanism that serves both internal DWS officials (who sign letters in the internal application) and external water users (who sign acknowledgements and responses through the public portal). As shown in Figure 4, the flow begins when a letter PDF is generated and ends when the signed document is stored, the letter issuance record is updated, and downstream notifications are triggered.

The signature mechanism is designed to produce legally admissible electronic signatures that comply with the Electronic Communications and Transactions Act (ECTA), Act 25 of 2002, by capturing sufficient metadata to establish the identity of the signatory, the integrity of the signed document, and the intent to sign.

6.2 Signature Request Creation

The signature flow is initiated when the LetterService generates a letter PDF and creates a SignatureRequest record:

  • DocumentId — FK to the generated letter document
  • RequestedSignatoryId — The user who is required to sign (identified by role and organisational unit)
  • Reason — The legal basis for the signature (e.g., "S35(1) Notice — Regional Manager approval")
  • Status — Initially set to Pending
  • CreatedDate — Timestamp of request creation
  • DocumentHashAtCreation — SHA-256 hash of the PDF at the time of request, used later for tamper detection

6.3 Notification Routing

The SignatureRequest triggers a notification through the NotificationService, routed based on the signatory type:

Internal User (DWS Official)

An in-app notification of type LetterReadyForSigning is displayed in the notification bell of the internal application. Clicking the notification routes the user to /Letters/{id}/Sign, where the letter PDF is displayed alongside the signature capture interface.

External Portal User (Water User)

A portal notification of type SignatureRequired is created, accompanied by an email notification to the external user's registered email address. The portal notification routes to /Portal/Cases/{id}/Sign, where the same signature capture interface is presented within the external portal layout.

6.4 Signature Capture

Both internal and external signing interfaces use SignaturePad.js to render an HTML5 canvas element where the user draws their signature. Upon submission, the following payload is assembled:

  • Signature Image — The drawn signature exported as a PNG data URI
  • Timestamp — The exact date and time of signing (server-validated)
  • IP Address — The signatory's IP address at the time of signing
  • User Agent — The browser/device information
  • Document Hash — SHA-256 hash of the document at the moment of signing, computed client-side and verified server-side
Tamper Detection
The document hash captured at signing is compared against the hash stored at request creation. If the hashes do not match, the signing is rejected, as it indicates the document has been modified after the signature was requested. This provides tamper-evidence as required by ECTA.

6.5 Verification and Storage

Upon submission of the signature payload, the server performs the following operations:

  1. Hash Verification — The submitted document hash is compared against the DocumentHashAtCreation on the SignatureRequest. Mismatches are rejected.
  2. DigitalSignature Record — A new DigitalSignature record is created, storing the signature image (PNG), the SHA-256 hash, the signatory's identity, IP address, user agent, and timestamp.
  3. PDF Embedding — The signature image is embedded into the letter PDF at the designated signature position, producing the final signed document.
  4. Document Storage — The signed PDF is stored (Azure Blob Storage or filesystem), and the UploadedDocument record is updated with the new BlobPath.
  5. LetterIssuance Update — The LetterIssuance record is updated with SignedDate, SignedById, DigitalSignatureId, and Status transitions to Signed.

6.6 Post-Signing

Once the letter is signed and the LetterIssuance record is updated, the following post-signing actions are triggered:

  • The NotificationService raises a LetterSignedEvent domain event, triggering notifications to relevant parties (the case validator, the water user, and any watchers on the case).
  • The LetterService.IssueLetter() method is invoked to dispatch the signed letter via the appropriate delivery method:
    • Post — Registered post with tracking number recorded in PostalRegistrationNumber
    • Email — Electronic delivery to the water user's registered email address with the signed PDF attached
    • In-Person — Flagged for hand delivery (mandatory for Letter 1); the serving official is recorded
    • Other — Alternative delivery methods as determined by the responsible authority
  • The workflow state transitions to the next appropriate state (e.g., L1_Signed → L1_Issued).

6.7 Legal Compliance

The electronic signature implementation is designed to comply with the Electronic Communications and Transactions Act (ECTA), Act 25 of 2002, which establishes the legal framework for electronic signatures in South Africa. Key compliance measures include:

  • Identity Association — Each signature is linked to an authenticated user account (internal via ASP.NET Identity, external via the separate portal identity regime with mandatory MFA).
  • Tamper Evidence — The DocumentHashAtSigning is permanently stored and can be compared against the current document hash at any future point. Any modification to the signed document is detectable.
  • Intent to Sign — The signing interface requires an explicit action (drawing a signature and clicking "Confirm"), with clear indication of what is being signed and the legal implications.
  • Audit Trail — The complete signing process is logged: request creation, notification delivery, signature capture (with IP, user agent, timestamp), verification, and PDF embedding. All audit records are immutable.
  • Non-Repudiation — The combination of authenticated identity, captured biometric-style signature image, IP address, device information, and timestamped hash chain provides strong evidence against repudiation.

7. Design Patterns Summary

The following table summarises the core design patterns employed across the workflow and process architecture:

Pattern Application Component
State Machine Governs the lifecycle of every FileMaster case through defined states and transitions. Ensures deterministic, auditable progression through control points. WorkflowEngine
Guard / Specification Each state transition is protected by composable guard objects that verify preconditions (data presence, role authorisation, business rules) before permitting advancement. ITransitionGuard
Repository Abstracts all data access behind interface contracts, enabling testability and separation of the data layer from business logic. IPropertyInterface, IAddress, IForestation, etc.
Strategy Letter delivery methods (Post, Email, In-Person, Other) are implemented as interchangeable strategies, selected at runtime based on the letter type and case requirements. LetterService delivery methods
Observer / Domain Events State transitions and significant actions publish domain events that are consumed by loosely coupled subscribers (notifications, audit logging, integration updates). NotificationService, AuditService
Mediator Engine coordination is managed through a central mediator, preventing direct dependencies between engines while enabling complex multi-engine workflows. Engine layer coordination
Immutable Log All audit entries are append-only records that can never be modified or deleted, providing a tamper-proof evidentiary trail for legal and regulatory compliance. AuditEntry table

The workflow and process designs documented here represent the target architecture for the DWA V&V System. The following implementation tasks are prioritised for the next development phases:
