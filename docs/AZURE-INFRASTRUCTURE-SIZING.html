1. Overview & Approach

This document specifies the minimum initial Azure deployment for the DWA V&V system using Azure App Service (no containerisation). The sizing targets an early-phase rollout (2–3 WMAs, ~50 concurrent internal users) with a clear scale-up path as more WMAs come online.

Region: South Africa North (Johannesburg) — the closest Azure region to DWS Head Office in Pretoria (~55 km). All services deployed in a single region to minimise latency and keep data within South African borders.

Why Azure App Service?

  • Fully managed PaaS — no OS patching, no VM administration
  • Built-in support for ASP.NET Core 10.0
  • Deployment slots for zero-downtime releases
  • Auto-scale available on Standard tier and above
  • TLS/SSL included with custom domain support

Why S1 over B1/B2? Basic tier lacks auto-scale, deployment slots, and custom domain SSL. Standard S1 provides these essentials at a manageable cost. If initial load proves light, B2 (R658/month) is a fallback option for development/staging environments.

Why S2 (50 DTU)? The V&V system has complex queries across 45+ entities with EF Core. S0 (10 DTU) will bottleneck on joins across Property → FileMaster → FieldAndCrop → Validation chains. S2 provides comfortable headroom for SAPWAT batch calculations, audit log writes, and concurrent report queries. S1 (20 DTU) is a viable fallback if budget is tight and batch processing is scheduled off-peak.

7. Scaling Path

Azure App Service and SQL Database scale without downtime. The recommended upgrade path as load increases:

TriggerActionAdditional Cost
CPU > 70% sustained or >100 concurrent usersScale App Service to S2 (2 vCPU, 3.5 GB RAM)+R 1,864/month
Auto-scale regularly hitting 2 instancesSet minimum instances to 2+R 1,877/month
DTU > 80% sustainedScale SQL to S3 (100 DTU)+R 1,720/month
All 9 WMAs live, >200 concurrent usersScale App Service to P1v3 (2 vCPU, 8 GB) + SQL S4 (200 DTU)Review at that point
Blob > 5 TBEnable lifecycle policies (Hot → Cool → Archive)Net cost reduction
Key advantage over on-premise: Scaling takes minutes, not procurement cycles. You can start minimal and right-size based on actual usage data from Application Insights.

Long-term retention (PAJA compliance): Extend SQL backup retention to 10 years using Long-Term Retention (LTR) policies. Cost: ~$0.015/GB/month for LTR storage. For a 30 GB database, this is approximately R 8/month — negligible.

9. Recommendation

We recommend deploying with the Standard S1 App Service + Standard S2 SQL Database configuration as specified above. This provides:

  • Immediate availability — deploy within hours, not weeks
  • Low initial cost — ~R 4,250/month (~R 51,000/year) vs R 500,000+ capital outlay for on-premise
  • No hardware administration — OS patching, SSL, backups all managed by Azure
  • Data sovereignty — South Africa North region keeps all data within SA borders
  • SLA-backed uptime — 99.95% for App Service, 99.99% for SQL Database
  • Elastic scaling — scale up or out in minutes as more WMAs go live
  • Built-in DR — automated SQL backups with 5-minute RPO at no extra cost
Pre-requisite: DWS requires an Azure subscription. If using Azure Government or an EA (Enterprise Agreement), pricing may differ from the pay-as-you-go rates quoted above. An EA typically provides 5–15% discount on listed prices.
